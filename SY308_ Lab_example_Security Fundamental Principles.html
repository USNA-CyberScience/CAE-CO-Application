Lab: Buffer Overflow Attack

Consider a program buggycat. The program repeats the following until the user types "exit":
  • Ask for the filename.
  • Output the contents of the file.
The sample execution is as follows: 
> echo -n 'hello'> file1
> echo -n 'world'> file2
> sudo sysctl -w kernel.randomize_va_space=0
> chmod 755 buggycat
> ./buggycat

] file1
hello
] file2
world
] exit
The program was obtained by compiling buggycat.c as follows: 
> gcc -g -fno-stack-protector -z execstack -o buggycat buggycat.c
Get a shell. Do so in GDB, and then without GDB. Make sure the same payload works both with and without GDB.
  1. (10 pts) Note the code contains the following statement: 
    scanf("%256s", filename);
    What does %256s mean?

    Hint: Check the reference manual. In particular, focus on the explanation of "width".

  2. (10 pts) Locate where the buffer-overflow vulnerability lies in the source code.
  3. (10 pts) Outline your attack strategy.
  4. (30 pts) Give a python script creating your payload.
  5. (20 pts) Justify why your payload should work. In particular, give a GDB log that you used to calculate the number of NOPs and the modified return address.
  6. (10 pts) Give an execution log showing that you obtained a shell in GDB.
  7. (10 pts) Give an execution log showing that you obtained a shell without GDB.
